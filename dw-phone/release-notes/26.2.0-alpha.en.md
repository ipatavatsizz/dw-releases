## [March 8, 2026] - This Time the Phone Is Not Just Pretty, It Actually Feels Like a System

Hey. I wanted this note to be longer, more honest, and more detailed on purpose, because from the outside this commit looks like "calling was added, messaging got cleaned up, camera was introduced," but the truth runs deeper than that. The biggest focus of this commit was not adding a few flashy features. It was reworking the backbone of the phone.

For a while, what kept bothering me was this: even when the interface looked good, too many pieces behind the scenes were still disconnected from each other. One part had grown past its original event model, another still relied on an old router, another had decent UI but incomplete data flow. Players do not always see that in logs, but they absolutely feel it. That is why this update leaned less on theory and more on real use. I tested the phone in its current state, saw where it fell apart, and got reminded once again that testing matters more than assumptions. So this commit is not just about adding features. It is about making the existing ones truly work, truly cooperate, and actually feel like parts of a real phone.

### 📞 Calling is now a complete feature, not a half-finished experiment

This is easily one of the biggest themes of the update. Since a lot of the original implementation was written together with AI, the calling system looked correct in theory. Then a friend actually tried it in practice, and it immediately exposed the weak spots. Thankfully, that same real-world testing helped us find the missing pieces quickly, close the gaps, and catch a number of issues that never showed up on paper. If there is one lesson here, it is that theory stays perfect only until the moment it meets reality.

Calling now works as a full flow: starting a call, showing an incoming call, answering, rejecting, ending it, timing out unanswered calls, and cleaning everything up properly when a player drops. On the server side, a new `CallService` now handles number validation, busy state checks, online player resolution, call bookkeeping, and voice channel management through `pma-voice`. In other words, calling is no longer an effect layered on top of the UI. It is a real system.

The UI side was not left behind either. `CallOverlay` was added, `Dynamic Island` can now switch into a compact active-call mode, and `NavigationBar` behaves differently while a call is in progress. The result is subtle, but important: when the phone is on a call, it finally feels occupied by something real.

### 💬 Messaging finally has a real data flow behind it

Messaging had a similar problem. In theory, it looked complete enough. In practice, once I tested it properly inside FiveM, it became obvious that some of the most important pieces had been left out: database wiring, reliable UI synchronization, and the actual server-backed conversation flow. That is all much more structured now.

From the outside, it might look like only a few screens changed. But one of the most important parts of this commit lives here. Messaging is no longer just a set of nice-looking conversation bubbles. It now has real conversation creation and reuse, participant management, message persistence, unread tracking, and live delivery to online players.

On the server, the new `MessageService` brings `getMessages` and `sendMessage` under one proper flow. The database side now works with conversations, participants, and messages as connected parts of the same structure. If a direct conversation already exists, it gets reused. If not, a new one is created. Sending a message does not just write a row anymore; it also updates unread state and emits live events to relevant online participants.

On the phone side, `Compose` and `Conversation` now work through the real SDK flow instead of the old "let's keep it local for now and fix it later" feeling. That gap has been closed in a meaningful way.

### 📷 The camera finally behaves like a camera

This was one of the parts I enjoyed building the most. Because adding a camera app is never just about placing a button on the screen. It means dealing with the in-game render, putting the player into the correct animation, separating selfie and rear camera behavior, handling capture flow, and saving the resulting media correctly into the gallery.

This commit built a serious foundation for camera support on both the FiveM side and the web side. On the client, scripted camera support was added, selfie animation got special handling, and focus/input behavior during capture was cleaned up. Screenshot upload is now bridged through `screencapture`, and the server receives the uploaded image and stores it in the gallery. On the phone side, a full `Camera` app was added with its own manifest, locale files, and render logic.

So the camera is no longer a placeholder box sitting in the corner waiting for future work. It is now a real product feature.

### 🧭 In-app navigation was rebuilt into something modern

The old routing model had started to show its age. Technically it was string-based, fragile, and full of gaps. The new system is still path-driven, but it is now type-safe and much closer to how real applications actually need to behave. Opening an app on a specific path, pushing parameters through navigation, or forcing an existing app instance onto a different screen used to be far more brittle than it should have been.

That is why `useInternalRouter` was removed and replaced with the new type-safe `useAppRouter`. Contacts, Messages, Settings, and the other route-driven apps were moved over to it. This was not just a cosmetic refactor. Deep linking, targeted screen opening, parameterized navigation, and back-stack behavior are all much more predictable now.

On top of that, `DeepLinkRouter` was introduced and the old rushed action system was removed. That means opening an app from a notification, navigating to a specific page, or supporting new URI schemes later is now much cleaner both internally and externally.

### 🎛️ We also revisited all the smaller pieces that shape the feel of the phone

This commit is not only about major systems. At the end of the day, what the player actually touches is feel, not architecture, so I also spent time on many smaller but critical details.

- `VolumeHUD` was reworked, improved, and effectively replaced with the new `SystemVolumeBar`. It is more responsive, better animated, and visually much cleaner. When you adjust the volume, it now feels like an intentional system component instead of a temporary overlay.
- The status bar was split apart into a modular structure. The old version had grown into a huge single file that was painful to maintain and even awkward for AI-assisted workflows. Icons, panel structures, control center pieces, and helper hooks are now separated into a cleaner system.
- A luminance-based readability system was added so the status bar can adapt to the brightness of the selected wallpaper. In other words, light wallpapers no longer wash out the top bar and dark wallpapers no longer bury it.
- A `minimal` variant was added to the toast system, which fits call and lock-screen scenarios much better.
- `NotificationItem` behavior changed so notifications are no longer just visual cards; they now act as proper entry points into apps, including support for the deeper linking structure.
- The `Recent Apps` dismiss threshold and motion behavior were refined. The overall interaction feels more natural now, and active-call behavior inside Recents is also better handled.

These things sound small on their own, but when they come together, they are exactly what makes the phone stop feeling like a mockup and start feeling like a product.

### 🧱 Shared types and contracts were expanded too

The `core` package also grew in an important way. New call types, new RPC payload and response definitions, new event names, and shared structures like `BaseAppProps` were added. That matters because the UI, client, and server are now speaking to each other with less guesswork and more formal contracts.

One of the most valuable parts of this commit is precisely this invisible side of it: the groundwork for future features is now stronger than it was before. New apps and new features can be added on top of a firmer base with less risk and better developer experience.

### 🛠️ Build and project structure got some cleanup too

`vite-plugin-svgr` was added for SVG handling. Type declarations needed by the camera and the new components were updated. Some outdated settings pages were removed, some behavior was moved into more appropriate top-level settings, and the browser-development mocks were revised so they better reflect the new flows. We are still committed to keeping browser-side development comfortable, because it makes UI iteration dramatically easier on our side.

### Final thought: this commit is not just a feature list, it is a level-up

If I had to summarize this update in one sentence, it would be this: with this commit, DW Phone moved from being a phone that looks good to being a phone that actually has engineering behind it.

Is there still more to do? Of course. But the road ahead is cleaner now. Adding new features is easier, breaking existing ones is harder, and building something that feels trustworthy to the player is much more realistic than it was before.

And honestly, for the first time in a while, I looked at the phone and felt not just "this looks nice," but "yes, this system is finally starting to settle into place."

## [March 1, 2026] - Pre-Alpha Architecture Overhaul & Details

Hey everyone. Before dropping the initial DW Phone 26.2.0-alpha release I've been working on for a while, I wanted to share what's been changing behind the scenes. Honestly, with this update, I focused on turning the phone from just a "working script" into a device you actually hold in your hands in-game. I ripped out everything in the infrastructure and UI that I found illogical. Here is what you can expect soon:

### ⚙️ Under the Hood: Escaping NUI & RPC Architecture

For me, this is the biggest deal of this process. The old NUI callbacks and standard event structures used to freak out, especially when things got heavy on the server. I moved the entire system to a solid **RPC (Remote Procedure Call)** architecture we wrote ourselves. Contacts, Messages, and session management now communicate flawlessly over this structure. I also closed the era of manual SQL execution for database updates; everything is handled automatically with `dw_phone_schema_migrations`.

### 📱 Core Apps: Contacts, Messages, and App Store

The old Contacts and Messages apps did the job but lacked soul. I completely redesigned both:

- **Contacts:** Added a built-in keypad and a real 'Recents' tab to help you remember who you talked to and when.
- **Messages:** Got a completely modern 'Inbox' design and proper group info pages.
- **App Store:** In the old system, we forced all apps onto everyone. Now we have a real market; install and delete what you want. Your phone, your rules.

### 🎨 Details I Obsessed Over: UI, Animations, and DND

I spent a lot of time on "small but effective" details to fix the feel:

- **Volume HUD:** When on the home screen, pressing the hardware volume keys (or shortcut) slides an iOS-style, smoothly animated bar from the edge of the screen.
- **Realistic Theme:** The auto-theme feature now relies on _in-game time_ (07:00 / 19:00), not your PC's clock. When it gets dark in the game, your phone goes into dark mode.
- **Control Center & DND:** I removed that cheap dark box in the background and added a real blur effect reflecting the world behind it. I cleaned up the unnecessary buttons in the header. You can long-press any toggle to go straight to its settings. Most importantly: **Do Not Disturb (DND) is no longer decorative.** When you turn it on, it actually swallows notifications, and since this state is saved to the database (`sessions` table), your setting persists even if you reconnect.
- **Smooth Exits:** When clearing notifications or hitting "Close All" in Recent Apps, cards don't just vanish instantly; they all leave the screen one by one with a satisfying slide animation (stagger/cascade).

### 🌍 The Language Issue & Goodbye to Useless Settings

The era of single-language phones is over. I integrated full Turkish and English support (via `react-i18next`). Adding new languages is now very easy. By the way, I completely deleted the "Region" setting, which did absolutely nothing and just cluttered the menu.

### ✅ What This Update Changes for You

In short:

- **For players:** the phone is more predictable. Calling/messaging is more stable, DND actually works, and UI interactions feel smoother.
- **For server owners:** automatic migrations and the RPC transition reduce maintenance overhead and random data inconsistency issues.
- **For developers:** the codebase is more modular; adding apps, translations, and feature extensions is now safer and faster.

### ⚠️ Alpha Note: Intentionally Incomplete Areas

This build ships as “solid foundation,” not “everything finished.” So:

- **Gallery** backend is ready, but the full user flow is not finalized yet.
- **Bank** is currently a placeholder; the real target is proper live economy integration.
- Some micro-animations and empty states will continue to be refined during the alpha cycle.

### 🔜 What’s Next

Next focus areas on the roadmap:

- Finalize messaging flow (stronger thread behavior + media flow).
- Bring the Gallery app to a fully user-facing state.
- Expand in-app performance instrumentation and optimize low-FPS scenarios.

That’s it from me for now. Until new release notes are published, I will keep updating the current note regularly like a development journal; you can track what’s new through the update dates.
